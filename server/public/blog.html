Authorization architecture

Why Phantom Auth0 became an authorization product.

The project stopped being about whether a browser agent could click, read, or call tools. The real question became who it was allowed to act for and where that authority should live.

Auth0-first Delegated access Approval-aware

The important shift was simple: keep browser execution local, then move identity, delegation, and approval into a hosted Auth0 surface the user can actually inspect.

Automation was never the hard part.

Once the extension could navigate pages and trigger external workflows, the interesting problem was no longer capability. It was authority. If a browser agent can reach outside the tab and act in a provider system, the user needs a clear answer to three questions: who the actor is, which external accounts are delegated, and when a sensitive action crosses a line that should require approval.

That is the reason Phantom Auth0 changed shape. The companion stopped being a side surface. It became the actual control plane.

Auth0 became the visible trust layer.

In this version, Auth0 is not just the way the user signs in. It is the place where authority is established, constrained, and explained.

  • Universal Login establishes the actor.
  • Connected Accounts show which provider relationships are delegated.
  • Runtime exchange makes access possible without embedding raw provider credentials in the extension.
  • Approval-aware execution creates a real boundary before high-risk writes run.

The product therefore reads differently. It is no longer “browser agent plus integrations.” It is an authorization surface for browser agents.

Different actions deserve different friction.

A secure agent should not treat every action the same way. Reads should remain fast. Drafts should remain reviewable. High-risk writes should pause when they would mutate external state.

That distinction is what makes approval meaningful instead of ornamental. The user only gets interrupted when the system is about to do something irreversible enough to matter.

The companion exists to answer trust questions quickly.

The hosted console is there to answer the questions that matter in a live evaluation:

  • Which actor is this local browser session paired to?
  • Which external accounts are connected right now?
  • What action is the agent asking to perform?
  • Does that request require approval before it executes?
  • What happened after the request completed or failed?

That is the product value. Delegation stops being hidden runtime state and becomes something a user can verify before trusting the system.

The shortest way to understand the build.

Pair the extension. Sign in through Auth0. Connect a provider account. Ask the agent to plan a delegated action. If the action is sensitive, approve it before it runs. Then inspect the result in the same surface.

That loop is the whole idea. The rest of the implementation exists to make that loop defensible.
