Capabilities

Not just agent automation. Delegated authority with user control.

The marketing site now tells the actual product story: local browser execution on one side, hosted identity and approval surfaces on the other.

Hosted Auth0 companion

A dedicated web surface for login, pairing, connected accounts, action review, and delegated status instead of burying authority inside the extension.

Connected Google and GitHub

Users can attach external accounts through Auth0 and let Phantom act through delegated provider access instead of raw third-party credentials.

Approval gates for high-risk actions

Reads and drafts can move fast. Sends, posts, creates, and other state-changing actions can stop for explicit user approval first.

Local browser agent stays local

The extension still owns page context, browser-native interaction, and voice UX. Auth0 becomes the authority layer instead of replacing the agent.

Token Vault-backed execution

Provider access is minted through Auth0 at runtime, so the product surface can show who is connected, what is allowed, and when approval is required.

Action history you can inspect

Every delegated path has a visible companion-side history so the user can understand what the agent did, what it asked to do, and what was approved.

Workflow

How the product actually works

The user journey is short and judge-friendly: pair the extension, connect an account, run a delegated action, and inspect the approval state.

1

Pair the extension

The local browser agent generates a pairing request and points the user to the hosted companion surface.

2

Sign in with Auth0

The companion establishes identity and becomes the place where connected accounts and delegated state are visible.

3

Connect provider accounts

Google and GitHub accounts are linked through Auth0 so Phantom can request delegated access when the user asks it to act.

4

Approve sensitive actions

Low-risk reads can complete immediately. Higher-risk create, send, or post actions can stop at an approval boundary first.

Architecture

A cleaner split between local execution and hosted authority

This is the main product idea behind the hackathon build. The extension remains the operator. Auth0 becomes the trust surface.

  • Identity lives in the hosted companion app instead of the extension carrying long-lived provider secrets.
  • Connected account state is visible in product UI, not hidden as background infrastructure.
  • Delegated actions are planned and executed through explicit scopes and runtime token exchange.
  • Approval boundaries can be inserted only where they matter, without breaking the entire agent flow.
  • The same surface shows action history, approval outcomes, and current connected providers.
Trust Model

Why this feels different from a normal browser agent

The product is intentionally opinionated about control boundaries. It treats delegated authority as part of the user experience, not as hidden plumbing.

Without the Auth0 layer

Agent products often collapse local context, third-party credentials, and execution authority into one runtime.

  • Provider tokens are hard to reason about
  • User approval is vague or absent
  • Cross-app state is hidden behind tool calls

With Phantom Auth0

The companion surface makes identity, delegation, and approval visible enough for a user and a judge to understand in one pass.

  • Connected accounts are explicit and user-facing
  • High-risk actions can pause for approval
  • Action history explains what happened and why
