Core principles

Authority becomes a product surface, not hidden middleware.

The branding, copy, and structure now reflect the actual value of this build: Auth0 is not decorative login chrome. It is the visible control plane that determines who the agent acts for, what accounts it can use, and when it must stop for human approval.

Local execution, hosted authority

The extension still performs browser-native work. The hosted companion owns identity, linked accounts, approval state, and the explanation layer around delegated actions.

Security model users can read

Connected account state, token-backed delegation, and action history are surfaced in product UI so the trust model is legible to users, reviewers, and judges.

Approval inserted where it matters

Not every action needs friction. The system distinguishes low-risk reads from drafts and high-risk writes, so the user only gets interrupted when authority actually changes external state.

Execution flow

The security path is short enough to demo and strict enough to trust.

Pair the extension, authenticate through Auth0, attach external accounts, and let the agent request delegated work through a surface built to expose identity and control boundaries.

1. Pair a local runtime to a hosted actor

The browser extension starts a pairing request and hands off to the companion. The hosted surface becomes the authoritative session anchor for that local agent.

2. Authenticate and attach external accounts

The user signs in through Auth0, links providers, and gets a visible record of which delegated accounts are currently active.

3. Evaluate action class before execution

Low-risk reads can run directly. Drafts can be previewed. High-risk sends, creates, and posts can trigger approval before the gateway touches external state.

4. Log the outcome back into the control surface

The companion exposes status, approval decisions, and execution outcomes so the trust model remains visible after the action completes.

Operator view

What the user should always be able to answer.

A security-focused agent product should make three things obvious: who the system is acting for, which accounts are delegated, and whether a sensitive action required approval.

Control questions

The hosted console is designed to answer the core operational questions without leaving the product surface.

  • Which identity is currently paired to the local agent runtime?
  • Which provider accounts are connected and what scopes are available?
  • Did the agent read, draft, or attempt to mutate external state?
  • Was approval required before the action crossed a high-risk boundary?
  • What happened after the action request completed or failed?
