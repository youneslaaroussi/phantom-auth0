What changes

Authority stops being hidden middleware.

The point of the product is not another browser demo. It is a visible trust layer where the user can understand who the agent acts for, which accounts are delegated, and when approval is required.

Identity lives in a hosted surface.

Auth0 establishes the actor and makes that state legible in the companion instead of burying trust inside the extension runtime.

Delegation becomes user-facing.

Connected accounts and runtime token exchange expose which external systems are attached before the agent uses them.

Approval happens where it matters.

Reads and previews can stay fast. High-risk writes can pause at a boundary the user can inspect and approve.

Architecture

One diagram, one idea.

The extension remains the operator. The companion becomes the control surface. Auth0 becomes the authority layer that mediates connected accounts and write-sensitive actions.

Pair a local runtime to a hosted actor.

The browser starts a pair request, then hands control to the hosted surface where the user can approve the binding and establish who the session belongs to.

Attach external accounts through Auth0.

The user sees which providers are connected and what delegated paths are available before the browser agent tries to use them.

Classify action risk before execution.

Read paths can complete immediately. Draft paths stay reviewable. Writes can stop for explicit approval before external state changes.

Return status to the same surface.

The outcome, approval state, and action history all land back in the companion so the user can understand what happened after the request ran.

Judge path

The demo loop is short, clear, and defensible.

Everything important fits in one evaluation path: pair the extension, sign in through Auth0, connect a provider, attempt a delegated action, and inspect whether approval was required.

Step 01
Pair the browser session

The extension generates a pending local request and points the user to the hosted surface.

Step 02
Authenticate and connect

The user signs in, links a provider account, and exposes delegated authority inside the control plane.

Step 03
Plan a delegated action

The gateway decides whether the action is a read, a previewable draft, or a write that should stop for approval.

Step 04
Review the outcome

The same companion shows the approval state, execution result, and recorded action history.
